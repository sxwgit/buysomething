# 采购管理系统 v2 优化建议

> 基于当前 v1 代码审查 + 业务场景分析，按优先级排列。

## 业务场景回顾

- 约 300 人、3 个部门，使用频率低，操作必须简单直观
- 数据透视要清晰，方便领导查看
- 管理员更新数据后，其他用户应能感知到数据变化
- Excel 导入为纯新增，不覆盖已有数据
- 附件（审批邮件）按月归档
- 根管理员可管理其他管理员，其余权限一致
- 安全性非重点关注，但不意味着可以有明显漏洞

---

## 一、必须修复的功能缺陷（P0）

### 1. DataTables JS 加载顺序错误

**文件**: `templates/base.html:76-77`

当前顺序：
```html
<script src="dataTables.bootstrap5.min.js"></script>  <!-- 先加载了样式层 -->
<script src="jquery.dataTables.min.js"></script>        <!-- 后加载了核心层 -->
```
bootstrap5 集成脚本依赖核心脚本，顺序反了。虽然目前没报错（浏览器容错），但可能导致部分功能异常或未来版本升级后崩溃。

**修复**: 交换两行顺序。

### 2. 更新/删除/批量操作可作用于已软删除的记录

**文件**: `routes/procurement.py:218, 253, 268`

`update_procurement` 和 `delete_procurement` 用的是 `get_or_404(pid)`，没有 `is_deleted=0` 过滤。管理员可能误编辑一条已删除的记录。批量更新同理。

**修复**: 改为 `Procurement.query.filter_by(id=pid, is_deleted=0).first_or_404()`，批量更新加 `.filter(Procurement.is_deleted == 0)`。

### 3. `ADMIN_SESSION_HOURS` 配置未生效

**文件**: `config.py:12` 定义了 `ADMIN_SESSION_HOURS = 2`，但 `routes/admin.py:75` 硬编码 `timedelta(hours=2)`。

**修复**: 登录时读取 `current_app.config['ADMIN_SESSION_HOURS']`。

### 4. 报表导出和设置页缺少权限校验

**问题**:
- `/api/reports/export` 任何人可导出全部采购数据 Excel
- `/settings` 页面虽然导航隐藏，但 URL 直接可访问（页面 HTML 会加载，只是 API 调用会 403）

**修复**: 报表导出加上 `is_admin()` 检查（或至少限制为登录后可用）。设置页在路由层加 `is_admin()` 校验，未授权时重定向到首页。

## 二、数据实时同步设计（P1 — 你的核心需求）

### 现状问题

当前所有页面都是"打开时加载一次，之后不变"。管理员新增/修改数据后，其他用户浏览器里看到的是旧数据，必须手动刷新页面。

### 推荐方案：轻量轮询 + 数据版本号

考虑到 300 人规模、使用频率低，不需要 WebSocket 的复杂度。推荐一个简洁可靠的方案：

**后端**:
1. 在 `DropdownOption` 表或新增一个 `SystemConfig` 表中维护一个全局 `data_version`（整数）
2. 每次采购数据发生写操作（增/改/删/导入），`data_version += 1`
3. 新增 API `GET /api/data-version`，返回当前版本号

**前端**:
1. 页面加载时获取一次 `data_version`
2. 每 30 秒轮询 `/api/data-version`，如果版本号变化了，自动刷新当前数据（表格重新加载、汇总重新请求）
3. 管理员自己操作后也会触发版本号变化，实现"自己操作后其他人能看到"

**优点**: 实现简单（约 50 行代码），对服务器压力极小（版本号查询是一次简单 SELECT），完全满足"管理员刷新后其他人能看到"的需求。

**备选方案**: 如果后续数据更新非常频繁，可以考虑 SSE（Server-Sent Events），但对当前场景过度设计。

## 三、管理员权限分级（P1 — 根管理员能力）

### 现状问题

所有管理员权限完全相同，任何管理员都能创建/删除其他管理员。

### 建议

在 `AdminUser` 模型中加一个 `is_root` 布尔字段：

```python
is_root = db.Column(db.Boolean, default=False)
```

**权限规则**:
- `is_root=True` 的管理员可以：创建管理员、删除管理员、查看管理员列表
- 所有管理员（包括非 root）可以：新增/编辑/删除采购数据、导入导出、上传附件、修改自己的密码
- 系统初始化时，第一个管理员自动为 root
- 至少保留一个 root 管理员（不能把自己删掉，类似于现有的 `count <= 1` 逻辑）

**涉及修改**:
- `models.py`: AdminUser 加 `is_root` 字段
- `routes/admin.py`: `list_admin_users`、`create_admin_user`、`delete_admin_user` 加 `is_root` 校验
- `templates/settings.html`: 管理员列表标记 root 身份，非 root 用户隐藏"添加/删除"按钮

## 四、附件管理优化（P1 — 审批邮件归档场景）

### 现状问题

1. 当前附件上传后只显示文件名列表，无法区分"审批邮件"和其他文件
2. 没有"归档"概念，只是按年月部门堆放
3. 附件文件名是 UUID，下载时用原名，但无法在列表中直观看到邮件主题

### 建议

**4.1 附件增加类型/描述字段**

在 `Attachment` 模型中加：
```python
attachment_type = db.Column(db.String(50), default='审批邮件')  # 审批邮件 / 其他
description = db.Column(db.String(500))  # 邮件主题或简要说明
```

上传表单增加一个"说明"输入框，默认类型为"审批邮件"。

**4.2 附件增加上传时的文件类型白名单**

虽然是内部系统，但建议限制为常见的文档和邮件格式：
```python
ALLOWED_EXTENSIONS = {'.pdf', '.doc', '.docx', '.xls', '.xlsx', '.msg', '.eml', '.zip', '.rar', '.png', '.jpg'}
```

**4.3 附件上传路径安全校验**

当前 `department` 参数直接拼入路径（`routes/attachment.py:100`），如果传入 `../` 会写到 uploads 目录之外。需要校验：
```python
if '/' in department or '\\' in department or '..' in department:
    return jsonify({'msg': '部门名称不合法'}), 400
```

## 五、代码结构清理（P2）

### 5.1 消除重复代码

**问题**: 管理员迁移逻辑在 `app.py:29-35` 和 `admin.py:10-19` 重复。`change_password` 路由在 `app.py:67-82` 而非 `admin.py` 蓝图中。`escapeHtml`/`formatCurrency` 在两个 JS 文件中重复定义。

**建议**:
- 迁移逻辑只保留在 `admin.py` 的 `ensure_admin_users()` 中，`app.py` 调用它即可
- `change_password` 移入 `admin.py` 蓝图
- 前端公共函数提取到 `static/js/common.js`，模板中统一引入

### 5.2 `is_admin()` 重复查询优化

**问题**: `current_admin_user()` 调用 `is_admin()`（查一次 AdminUser），然后自己又查一次 AdminUser，加上 `is_admin()` 内部的 expire 检查，一次权限验证触发 2-3 次 DB 查询。

**建议**: 合并为一个函数，只查一次数据库：
```python
def current_admin_user():
    if 'is_admin' not in session:
        return None
    # ... expire check (无 DB 查询) ...
    user = AdminUser.query.get(session.get('admin_user_id'))
    if not user:
        session.clear()
    return user
```
其他地方需要判断是否管理员时，用 `current_admin_user() is not None` 代替 `is_admin()`。

### 5.3 报表 overview 性能优化

**文件**: `routes/report.py:90`

当前 `records = base.all()` 将全年所有采购记录加载到 Python 内存再循环统计。如果数据量增长到几千条，这会变慢。

**建议**: 改用 SQL 聚合（`func.count`/`func.sum`/`case`），一次查询出结果，和 `department_summary` 保持一致的模式。改动量不大但效果显著。

### 5.4 DataTables drawCallback 去掉冗余 API 调用

**文件**: `static/js/procurement.js:193-201`

每次表格重绘（包括翻页）都调用 `checkAdmin()` 请求服务器。管理员状态在页面加载时和登录/登出时已经确定，不需要每次翻页都重新检查。

**建议**: 移除 `drawCallback` 中的 `checkAdmin()` 调用，只在页面初始化和 `onAdminChange` 回调中更新。

## 六、用户体验提升（P2）

### 6.1 采购列表：进入页面默认显示当年数据

当前页面打开时没有任何筛选条件，加载全部历史数据。对于 300 人的使用场景，用户最关心的通常是当年数据。

**建议**: 页面初始化时，默认选中当前年份，这样打开页面就是当年的采购记录，加载更快、数据更聚焦。

### 6.2 采购列表：筛选后 KPI 卡片标题联动

当前 KPI 卡片标题固定为"当前记录数"/"当前总金额"，但"当前"指什么不明确。

**建议**: KPI 卡片标题动态反映筛选条件，例如"2026年记录数"/"行政部总金额"，让领导一眼能看懂数据口径。

### 6.3 报表页：默认加载当年数据

和采购列表一样，报表页打开时需要手动点"查询"才出数据。

**建议**: 页面加载后自动查询当年数据。

### 6.4 导入反馈优化

当前导入成功后只显示文字提示。用户无法确认具体导入了哪些数据。

**建议**: 导入完成后显示简要的结果汇总（成功 N 条、失败 N 条、失败原因列表），并且在设置页提供一个"查看最近导入记录"的入口。

### 6.5 导航栏高亮当前页面

当前已经用 `active_page` 实现了导航高亮，这点做得不错。但"系统设置"在非管理员时完全隐藏，可以考虑对非管理员显示一个灰色的"设置"入口，点击后提示需要管理员权限，让用户知道这个功能存在。

## 七、安全相关（仅列关键项，优先级低）

以下项目按你的场景安全优先级不高，但记录在此供参考：

| 项目 | 说明 | 建议 |
|------|------|------|
| SECRET_KEY 硬编码 | `config.py` 中有默认值 | 部署时通过环境变量设置即可 |
| 无登录限流 | 暴力破解风险 | 内网场景风险低，如需加可用 `Flask-Limiter` |
| 无 CSRF 防护 | Cookie-based auth 无 token | 内网场景可接受 |
| CDN 无 SRI | 前端资源可被篡改 | 内网场景风险低 |

唯一需要关注的是 **附件上传路径穿越**（5.3 中已提到），这个应该修复。
