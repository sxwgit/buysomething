# 采购管理模块走读、问题清单与重构建议

## 1. 模块概览

当前模块是一个基于 `Flask + SQLAlchemy + Jinja2 + jQuery/DataTables` 的采购管理系统，采用服务端模板渲染配合少量前端脚本实现交互。

主要目录结构如下：

- `app.py`：应用入口，负责初始化 Flask、注册蓝图、建表和创建索引
- `models.py`：核心数据模型定义
- `routes/`：按业务拆分的后端接口
- `templates/`：Jinja2 页面模板
- `static/js/`：前端页面脚本
- `static/css/custom.css`：全局样式
- `init_db.py`：数据库初始化和默认数据灌入脚本

## 2. 核心运行链路

### 2.1 应用启动

入口在 `app.py`：

- 创建 Flask 应用并加载 `Config`
- 初始化 `SQLAlchemy`
- 注册以下蓝图：
  - `procurement_bp`
  - `attachment_bp`
  - `report_bp`
  - `admin_bp`
  - `settings_bp`
  - `data_import_bp`
- 启动时自动执行：
  - `db.create_all()`
  - `ensure_admin_users()`
  - 为采购和附件表建立若干查询索引

页面入口路由：

- `/`：采购列表页
- `/attachments`：附件管理页
- `/reports`：透视报表页
- `/settings`：系统设置页，仅管理员可见

### 2.2 数据模型

`models.py` 中的主要表：

- `Procurement`
  - 采购主表
  - 包含年月、资产分类、物品信息、部门、申请人、预算数量、单价、总价、状态等字段
  - `is_deleted` 用于逻辑删除

- `Attachment`
  - 附件表
  - 记录附件所属年月、部门、原始文件名、保存文件名、大小、说明等

- `DropdownOption`
  - 通用下拉配置表
  - 用于资产分类、部门、采购状态等可配置枚举

- `AdminPassword`
  - 老版本管理员密码表，当前主要用于兼容迁移

- `AdminUser`
  - 新版管理员账号表
  - 支持用户名、密码哈希、根管理员标识

- `DataVersion`
  - 数据版本表
  - 用于前端轮询侦测数据变化并刷新页面

### 2.3 页面与接口关系

#### 采购列表页

模板：`templates/index.html`  
脚本：`static/js/procurement.js`

主要能力：

- 多条件筛选：年份、月份、部门、状态、关键字
- DataTables 服务端分页
- 顶部 KPI 汇总卡片
- 新增、编辑、删除采购记录
- 行内编辑数量、单价、原因、备注、状态
- 批量更新状态
- 采购详情弹窗

主要接口：

- `GET /api/procurements/filter-metadata`
- `GET /api/procurements`
- `GET /api/procurements/summary`
- `GET /api/procurements/<id>`
- `POST /api/procurements`
- `PUT /api/procurements/<id>`
- `DELETE /api/procurements/<id>`
- `POST /api/procurements/batch-status`

#### 附件管理页

模板：`templates/attachments.html`  
脚本：`static/js/attachment.js`

主要能力：

- 按年月筛选附件
- 展示各部门月度附件上传完整性
- 附件上传、下载、删除

主要接口：

- `GET /api/attachments`
- `GET /api/attachments/coverage`
- `POST /api/attachments/upload`
- `GET /api/attachments/<id>/download`
- `DELETE /api/attachments/<id>`

#### 透视报表页

模板：`templates/reports.html`  
脚本：`static/js/report.js`

主要能力：

- 年度或月度看板
- 部门采购汇总
- 部门金额占比
- 资产分类分布
- Excel 导出

主要接口：

- `GET /api/reports/overview`
- `GET /api/reports/department-summary`
- `GET /api/reports/department-ratio`
- `GET /api/reports/category-distribution`
- `GET /api/reports/export`

#### 系统设置页

模板：`templates/settings.html`

主要能力：

- 下拉选项维护
- Excel 导入导出
- 当前管理员改密
- 根管理员管理其他管理员

主要接口：

- `GET /api/options/<category>`
- `POST /api/options`
- `PUT /api/options/<id>`
- `DELETE /api/options/<id>`
- `GET /api/admin/users`
- `POST /api/admin/users`
- `DELETE /api/admin/users/<id>`
- `POST /api/admin/change-password`
- `GET /api/import/template`
- `GET /api/import/export-data`
- `POST /api/import/excel`

### 2.4 权限机制

管理员认证逻辑集中在 `routes/admin.py`：

- 登录成功后将管理员状态写入 `session`
- 会话带过期时间
- 普通管理员可维护业务数据
- 根管理员可维护管理员账号

前端通过：

- `GET /admin/check`
- `POST /admin/login`
- `POST /admin/logout`

来同步管理员状态，并控制 `.admin-only` 与 `.root-admin-only` 元素显隐。

### 2.5 实时刷新机制

系统使用 `DataVersion` 做轻量级前端轮询同步：

- 前端每 30 秒请求一次 `GET /api/data-version`
- 采购变更后调用 `bump_data_version()`
- 页面检测到版本变化时触发各自的刷新函数

这个方案简单直接，适合当前规模。

## 3. 当前问题清单

以下问题按优先级分组。

### 3.1 高优先级问题

1. `AdminPassword` 的密码方法定义位置明显错误

- 在 `models.py` 中，`set_password` 和 `check_password` 被写在了 `DataVersion` 类里
- 但 `init_db.py` 中实际调用的是 `AdminPassword().set_password(...)`
- 这意味着初始化脚本在某些场景下会直接报错

2. 独立管理员登录页与当前登录接口协议不一致

- `templates/admin_login.html` 仍然只提交 `password`
- `routes/admin.py` 的 `/admin/login` 现在要求 `username + password`
- 这会导致独立登录页不可用

3. 前端页面对接口错误处理不统一

- 一部分使用 `alert`
- 一部分直接忽略错误
- 一部分展示在页面内
- 用户体验不一致，也不利于后续统一维护

4. 前端交互过于分散，页面脚本文件偏长

- `static/js/procurement.js` 已经承担筛选、分页、弹窗、行内编辑、批量操作等多种职责
- 继续叠加需求时，维护复杂度会迅速上升

### 3.2 中优先级问题

5. 页面样式较旧，信息层级与视觉引导不足

- 配色偏默认 Bootstrap 风格
- 卡片、筛选区、表格、分页器缺少统一的现代化视觉语言
- 页面第一眼可读性和专业感偏弱

6. DataTables 的分页交互生硬

- 首页、末页、上下页、页码、跳转区全部并列
- 信息密度高但不美观
- 跳页控件侵入感强

7. 设置页内嵌大量脚本

- `templates/settings.html` 直接包含较长的 `<script>`
- 不利于分层，也不便于后续复用和测试

8. 报表页导出按钮未做管理员限制

- 后端 `export` 接口要求管理员
- 前端按钮对所有用户可见
- 非管理员点击后会收到接口报错，体验不够平滑

9. 采购筛选和表单配置来源依赖下拉配置，但缺少缓存与统一加载层

- 现在由各页面各自请求 `/api/options/...`
- 会造成逻辑重复

### 3.3 低优先级问题

10. 使用浮点数存储金额有精度风险

- `unit_price`、`total_price` 使用 `Float`
- 当前金额规模可能问题不明显，但长期看更适合使用定点类型

11. 软删除与附件物理删除策略不统一

- 采购采用逻辑删除
- 附件采用数据库删除 + 文件物理删除
- 策略上可以更明确一些

12. 代码中存在历史兼容残留

- `AdminPassword` 与 `AdminUser` 双模型并存
- 登录页面与登录方式发生过切换
- 说明该模块已有迭代痕迹，但清理未完成

## 4. 重构建议

### 4.1 第一阶段：低风险整理

建议优先做以下不影响业务能力的整理：

1. 修复管理员密码模型错误

- 将 `set_password` / `check_password` 正确归属到 `AdminPassword`
- 保留兼容迁移逻辑，但避免未来初始化失败

2. 修复独立管理员登录页

- 与当前 `/admin/login` 协议保持一致
- 或者如果该页已不再需要，明确废弃并移除入口

3. 拆分前端脚本

- 将 `procurement.js` 拆成：
  - 筛选与查询
  - 表格渲染
  - 表单弹窗
  - 行内编辑
  - 批量操作

4. 统一消息提示组件

- 用页面级 toast / alert 组件替换分散的 `alert()`
- 统一成功、失败、警告、加载中状态

### 4.2 第二阶段：前端结构优化

在不替换后端的前提下，可以先把前端体验升级到更现代的形态：

1. 统一视觉系统

- 建立 CSS 变量
- 统一颜色、阴影、边框、圆角、间距
- 重做顶部导航、筛选条、KPI 卡片、表格、分页器

2. 重做分页体验

- 保留服务端分页能力
- 改为更简洁的分页器样式
- 弱化“首页/末页/跳页输入”这些高噪声控件

3. 调整页面布局

- 给每个页面增加更明确的标题区和说明区
- 将“筛选区、摘要区、数据区”层次拉开

4. 设置页脚本外移

- 将内联 JS 提取到 `static/js/settings.js`

### 4.3 第三阶段：中期架构改造

如果后续页面交互继续增加，建议考虑引入现代前端框架，但应采取渐进式改造：

1. 推荐方案

- 保留 Flask 后端接口
- 新增独立前端层，例如 `Vue 3 + Vite` 或 `React + Vite`
- 先从采购列表页改造，再逐步替换附件页、报表页、设置页

2. 不建议一次性全量替换

- 当前系统已可用
- 一次性迁移风险高，容易影响现有录入、查询、导入导出功能

3. 更适合先改造的页面

- `采购列表页`
- 原因：
  - 交互最复杂
  - 行内编辑和表格状态管理最适合组件化
  - 改造收益最高

### 4.4 数据与后端建议

1. 金额字段建议逐步迁移为定点数

- 如 `Numeric(12, 2)`

2. 下拉配置建议增加唯一约束

- 避免同一 `category + value` 被重复录入

3. 采购状态建议枚举化

- 目前状态依赖下拉配置与前端文案约定
- 长期看容易出现脏数据

4. 报表查询可以进一步收敛

- 一些 `IN (subquery)` 的写法可以直接复用基础过滤条件
- 未来数据量变大后可再优化 SQL 结构

## 5. 结论

这个模块当前的优点是：

- 业务链路完整
- 接口边界清晰
- 对中小规模场景足够直接
- 使用门槛低

当前最值得优先处理的不是立刻重写，而是：

1. 修掉明显的历史兼容问题
2. 提升页面视觉与交互体验
3. 在保持接口稳定的前提下整理前端结构

如果后续需求继续增长，再考虑用现代前端框架做渐进式替换，会比现在直接推翻重来更稳妥。
