# 采购管理系统 LLM 开发执行手册

## 1. 文档用途

这不是一份泛泛的项目计划，而是一份可以直接交给大模型持续执行的开发手册。

目标只有三个：

1. 明确每个阶段到底修什么
2. 明确每个阶段做完后怎么测试、怎么整理 git
3. 明确大模型在开发时哪些事可以做，哪些事不能做

## 2. 项目定位

本系统是一个小部门内部使用的采购跟踪网站，不是正式的采购执行平台，也不是复杂审批系统。

系统服务的核心场景只有四个：

- 收集采购需求
- 做基础预审
- 给领导查看
- 做数据整理和简单透视

所有开发动作都必须围绕这四个场景展开。凡是明显偏向“复杂流程引擎、多角色审批系统、正式采购 ERP”的方向，都不在本轮范围内。

## 3. 开发总策略

### 3.1 总体原则

1. 按阶段推进，每个阶段只解决一类问题。
2. 每个阶段必须有明确的开始条件、完成标准和验收动作。
3. 每个阶段结束后必须先测试，再整理 git，再进入下一阶段。
4. 所有改动只做本地提交，不推送到 GitHub。
5. 发现新问题时，先判断是否属于当前阶段；不属于就记入后续阶段，不顺手扩散。

### 3.2 优先级原则

开发优先顺序固定如下：

1. 先修“会误事”的问题
2. 再修“会费劲”的问题
3. 最后修“看不清”的问题

对应到本项目，就是：

1. 先修列表查询、编辑、导出这类高频故障
2. 再优化首页列表的可读性和预审效率
3. 再补轻量业务看板
4. 最后补附件完整性视图

### 3.3 范围边界

本轮明确不做：

- 多级审批流
- 多角色权限系统
- 复杂消息通知
- 预算控制系统
- 正式供应商管理
- 独立移动端
- 大规模 BI 平台化改造

如果开发过程中出现“这个功能也许以后会需要”，默认先不做，除非它直接影响当前四个核心场景。

## 4. 每阶段统一执行流程

每个阶段必须严格走完下面 7 步，不能跳步。

### 第 1 步：确认阶段边界

开始开发前先确认：

- 当前只做哪一个阶段
- 本阶段允许改哪些文件
- 本阶段不能顺手扩展哪些内容

### 第 2 步：阅读相关代码

至少先阅读：

- 对应模板文件
- 对应前端脚本
- 对应后端路由
- 如有必要再补读模型或配置文件

禁止凭空重写；必须先基于现有实现理解问题，再做增量修补。

### 第 3 步：实施改动

改动时遵守：

- 优先小步修改
- 不混入无关重构
- 不把多个阶段目标揉在一起

### 第 4 步：自测

本阶段改完后必须先做自测，至少覆盖：

- 基础页面可打开
- 涉及接口可用
- 本阶段目标功能有效
- 原有关键功能未明显回归

### 第 5 步：人工验收口径检查

除了代码正确，还要对照“业务是否真的更顺”做一次判断：

- 是否更方便收集采购数据
- 是否更方便做基础预审
- 是否更方便给领导看
- 是否更方便整理数据

如果只是“技术上完成”，但业务上并没有更顺，视为本阶段未完全达标。

### 第 6 步：整理 git

每阶段完成后只做本地整理：

```bash
git status --short
git diff --stat
git add <本阶段相关文件>
git commit -m "<本阶段提交信息>"
```

禁止动作：

- 不要 `git push`
- 不要 `git reset --hard`
- 不要把多个阶段混成一个提交
- 不要把无关文件混入本阶段提交

### 第 7 步：输出阶段结果

每轮结束时，大模型必须输出：

- 本阶段完成了什么
- 改了哪些文件
- 做了哪些测试
- 是否已经完成本地 git 提交
- 还剩什么风险或后续事项

## 6. 分阶段开发方案

---

### 阶段一：修复高频操作问题

阶段目标：

- 先保证系统“查得到、改得动、导得对”。

本阶段包含：

- 修复采购列表页部门筛选错误绑定问题
- 修复采购列表页状态筛选未加载问题
- 修复编辑旧记录时只查第一页数据的问题
- 修复报表页“部门占比”导出错导成“部门汇总”的问题
- 补上必要的前后端错误提示，避免操作失败但界面无反馈

本阶段不包含：

- 首页大改版
- 新增业务看板
- 重做附件页
- 重构全部前后端结构

建议修改文件：

- `static/js/procurement.js`
- `routes/procurement.py`
- `static/js/report.js`
- `routes/report.py`

完成标准：

1. 列表页按部门筛选结果正确
2. 列表页按状态筛选结果正确
3. 关键字、年份、月份、部门、状态可组合查询
4. 编辑任意记录都能正确回填和保存
5. 报表页三个导出入口都导出正确内容
6. 失败场景下页面有明确提示

测试动作：

1. 用 `test_client` 验证 `/api/procurements` 的组合筛选
2. 用 `test_client` 验证报表导出接口返回正常
3. 手工检查列表页筛选、编辑、导出路径
4. 回归检查首页、报表页、附件页能正常打开

git 收口：

```bash
git status --short
git diff --stat
git add static/js/procurement.js routes/procurement.py static/js/report.js routes/report.py
git commit -m "fix: repair procurement filters and report export flow"
```

---

### 阶段二：优化列表页可读性和预审效率

阶段目标：

- 让采购列表从“网页版 Excel”变成“更适合跟踪和预审”的工作台。

本阶段包含：

- 优化列表字段展示层级，突出部门、物品、金额、状态、申请原因
- 处理长文本展示问题，避免申请原因和备注一眼看不清
- 适当减少一屏信息压迫感，让常用信息更容易扫读
- 保持轻量，不引入复杂表单流程

本阶段不包含：

- 新增审批流
- 新增复杂详情页体系
- 改造数据库结构

建议修改文件：

- `templates/index.html`
- `static/js/procurement.js`
- `static/css/custom.css`

完成标准：

1. 首页主要信息能在一屏快速看清
2. 长申请原因不会把表格挤坏，也不会完全看不到
3. 常用筛选、编辑、查看动作不受影响
4. 窄屏下页面不会严重错位

测试动作：

1. 手工检查首页一屏信息密度
2. 验证长文本、空文本、金额字段展示是否正常
3. 回归测试筛选、编辑、状态显示
4. 检查桌面宽屏和较窄窗口下的布局

git 收口：

```bash
git status --short
git diff --stat
git add templates/index.html static/js/procurement.js static/css/custom.css
git commit -m "feat: improve procurement list readability for review workflow"
```

### 阶段三：补一个轻量业务看板

阶段目标：

- 让系统能支持“预审”和“给领导看”，而不只是统计金额。

本阶段包含：

- 在报表页或首页上方补充轻量业务摘要
- 增加按状态的汇总视图
- 增加“待处理”视角，而不是只有“按部门/分类统计”
- 保持轻量，不做复杂 BI，不做审批流程引擎

推荐看板内容：

- 当前筛选范围内总记录数
- `已申请 / 采购中 / 已完成` 各自数量和金额
- 本月待跟进条目数
- 金额最高的部门或高金额条目概览

本阶段不包含：

- 多页大屏驾驶舱
- 复杂钻取分析
- 独立权限看板

建议修改文件：

- `templates/reports.html`
- `static/js/report.js`
- `routes/report.py`
- 视需要调整 `templates/index.html`

完成标准：

1. 管理者打开报表页后，能在 10 秒内看懂当前采购情况
2. 能快速区分“待处理”和“已完成”
3. 汇总数据与列表明细能对得上
4. 年/月筛选后，看板数字同步变化

测试动作：

1. 用 `test_client` 验证新增汇总接口
2. 手工验证看板数字、筛选联动、展示文案
3. 对照列表明细抽查统计结果
4. 回归验证现有部门汇总、占比图、分类分布仍正常

git 收口：

```bash
git status --short
git diff --stat
git add templates/reports.html static/js/report.js routes/report.py
git commit -m "feat: add lightweight procurement business dashboard"
```

### 阶段四：补强附件完整性视图

阶段目标：

- 让附件页真正服务“每个月、每个部门有没有传审批邮件”的检查动作。

本阶段包含：

- 附件页增加“缺失视角”
- 让用户能快速看出某个月哪些部门已上传、哪些未上传
- 尽量复用现有部门下拉配置，不单独维护第二套名单

推荐实现方式：

- 保留当前附件列表
- 在列表上方或旁边增加“本月部门上传状态”
- 明确标出“已上传 / 未上传 / 附件数量”

本阶段不包含：

- 复杂文件审批流程
- 自动邮件解析
- 大规模附件审计系统

建议修改文件：

- `templates/attachments.html`
- `static/js/attachment.js`
- `routes/attachment.py`
- 视需要复用 `routes/settings.py`

完成标准：

1. 选择某年某月后，能直接看到各部门是否上传附件
2. 已上传的部门能继续查看附件明细
3. 没上传的部门能被明确标识出来
4. 上传、删除附件后，状态视图同步更新

测试动作：

1. 用 `test_client` 验证附件接口和部门状态数据
2. 手工验证已上传、未上传、空月份三种场景
3. 验证上传和删除后的刷新效果
4. 回归验证原有附件下载功能

git 收口：

```bash
git status --short
git diff --stat
git add templates/attachments.html static/js/attachment.js routes/attachment.py
git commit -m "feat: add attachment completeness view by department"
```

## 7. 每阶段统一测试矩阵

每个阶段结束后，测试至少覆盖下面四类。

### 7.1 语法和基础可运行性

```bash
python3 -m compileall app.py routes models.py
python3 -c "from app import app; c=app.test_client(); print(c.get('/').status_code); print(c.get('/reports').status_code); print(c.get('/attachments').status_code)"
```

### 7.2 接口烟雾测试

按当前阶段补充对应接口验证，不要只测首页。

建议格式：

```bash
python3 -c "from app import app; c=app.test_client(); print(c.get('<接口地址>').status_code)"
```

### 7.3 关键页面手工验证

至少检查：

- 首页
- 报表页
- 附件页
- 本阶段涉及的所有关键路径

### 7.4 回归检查

至少回答下面问题：

- 之前能用的筛选现在还好吗
- 之前能用的导出现在还好吗
- 之前能用的上传/下载现在还好吗
- 是否出现新的明显报错或空白区块

## 8. 每阶段统一 git 规范

### 8.1 进入阶段前

先看当前状态：

```bash
git status --short
git branch --show-current
```

如果工作区里有与本阶段无关的改动，先暂停并确认，不要直接混做。

### 8.2 结束阶段后

统一执行：

```bash
git status --short
git diff --stat
git add <本阶段文件>
git commit -m "<本阶段提交信息>"
```

### 8.3 提交信息规则

建议格式：

- `fix: ...` 用于修复错误
- `feat: ...` 用于增加轻量功能
- `refactor: ...` 仅用于必要的小范围重构

本项目当前推荐提交信息：

- `fix: repair procurement filters and report export flow`
- `feat: improve procurement list readability for review workflow`
- `feat: add lightweight procurement business dashboard`
- `feat: add attachment completeness view by department`

## 10. 推荐执行节奏

推荐严格按下面顺序推进：

1. 先做阶段一，确保系统基础可用
2. 阶段一通过后再做阶段二，优化日常操作体验
3. 阶段二通过后再做阶段三，补强领导查看和预审视角
4. 最后做阶段四，把附件检查闭环补齐

## 11. 最终验收标准

全部阶段完成后，系统至少要达到下面四个结果：

1. 采购数据可以稳定收集、查询、编辑、导出
2. 首页列表比当前明显更适合预审和跟踪
3. 报表页能同时满足统计和领导快速查看
4. 附件页能快速识别每月各部门是否已提交材料

如果某次开发结果没有让这四件事更顺，就说明开发偏离了本手册的目标，需要回到阶段边界重新收敛。
